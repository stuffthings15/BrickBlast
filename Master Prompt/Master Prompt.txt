# 1. Project Identity and Declaration

## 1.1 Project Name

Use the project name:

**BrickBlast: Velocity Market**

A subtitle may be used in marketing materials:

**A fast 2D brick breaker arcade game with progression, customization, lightweight online sync, and a points-based marketplace.**

## 1.2 Project Vision

Create a 2D brick blast / brick breaker game designed for short mobile play sessions. The player launches a ball or projectile sequence to destroy blocks, survive increasingly difficult levels, earn points, and spend those points in an in-game marketplace on visual and gameplay-adjacent upgrades.

The game should target the solo mobile player who wants an easy-to-understand arcade experience that can be played while waiting in line, sitting at the DMV, doing laundry, commuting, or passing a few idle minutes. The design must prioritize quick startup, responsive input, fast screen transitions, satisfying feedback, and low-friction replay.

The game must feel complete rather than like a tutorial clone. It must include menus, level flow, clear success and fail states, scoring, progression, upgrade purchasing, local persistence, visual feedback, audio feedback, game balance, and a final demo path.

## 1.3 Required Player Experience

The player should be able to:

Open the game and reach the main menu quickly.

Start a new session within one or two taps.

Control the paddle, aim, launcher, or ball system with responsive input.

Break bricks using predictable physics or deterministic projectile movement.

Earn points by clearing bricks, completing levels, achieving combos, or surviving streaks.

Lose when the ball or projectile objective fails according to the game rules.

Win a level when all required bricks are destroyed.

Advance through multiple levels with increasing complexity.

Spend earned points in a marketplace.

Equip or apply purchased cosmetics or gameplay modifiers.

See clear UI feedback for score, lives, level, currency, equipped item, and pause state.

Return later and retain progress using a persistence layer.

Play without network access, while optionally syncing when online.

View credits, instructions, and version information.

## 1.4 Final Build Philosophy

This project must ship as a polished final worthy of public presentation or publishing. It must not look like a raw classroom prototype. Every visible screen must feel intentional. Every major system must either be functional or clearly marked as intentionally scoped.

The final build must include:

A complete gameplay loop.

Original mechanics or progression layers beyond the starter scaffold.

Object-oriented structure in the codebase.

A marketplace / in-app transaction style system using earnable in-game currency.

A low-latency server sync or networking component.

A publication-ready build surface.

A short video trailer.

Professional documentation.

Architecture diagrams.

Testing log.

Submission checklist evidence.

# 3. Game Design Specification

## 3.1 Core Genre

This is a 2D brick blast / brick breaker game.

The core loop:

Aim or position
Launch or deflect ball
Destroy bricks
Earn points
Survive
Complete level
Earn currency
Upgrade or customize
Repeat at higher difficulty

## 3.2 Core Gameplay Loop

Implement this loop:

The player starts on the main menu.

The player taps Play.

The game loads the current available level.

The ball starts docked, paused, or ready to launch.

The player launches the ball.

The ball collides with bricks, paddle, walls, and possibly powerups.

Each brick hit reduces durability or destroys the brick.

Destroyed bricks grant score and sometimes currency.

The level ends in success when all required bricks are destroyed.

The level ends in failure when the player loses all lives, misses all balls, or fails the objective.

On success, show a results screen with score, currency earned, bonuses, and next-level button.

On failure, show retry, marketplace, and main menu options.

Between levels, allow the player to enter the marketplace and spend points.

Progress is saved.

## 3.3 Input Requirements

The game must be mobile-first but playable on desktop for testing.

Support:

Touch drag for paddle movement.

Mouse drag for desktop testing.

Optional keyboard support for debugging.

Tap to launch ball.

Tap pause button.

Tap marketplace buttons.

Tap settings toggles.

Input should feel responsive. Avoid delayed controls.

## 3.4 Camera and Screen Layout

Use a portrait-oriented mobile-friendly layout unless the starter project strongly favors landscape. The user prefers landscape video output, but gameplay may remain mobile-first. If gameplay is portrait, the video trailer should still be exported in landscape format with appropriate framing, side panels, or captured widescreen presentation.

Gameplay screen must show:

Score
Currency earned this run
Lives or balls remaining
Current level
Pause button
Active powerup indicator if applicable
Clear playfield boundaries
Readable bricks
Readable paddle and ball

## 3.5 Brick System

Bricks must be more than static rectangles. Implement multiple brick types.

Required brick types:

Standard Brick
Durable Brick
Reward Brick
Hazard Brick
Shielded Brick or Moving Brick

Suggested behavior:

Standard Brick: breaks in one hit.

Durable Brick: requires multiple hits and changes visual state.

Reward Brick: grants bonus currency or powerup chance.

Hazard Brick: punishes careless play, maybe speeds ball, damages paddle, or spawns obstacle.

Shielded Brick: requires hitting nearby key brick first, or takes reduced damage until exposed.

Moving Brick: moves horizontally or vertically within a lane.

Each brick should expose properties such as:

Health
ScoreValue
CurrencyReward
BrickType
IsRequiredForCompletion
HitEffect
DestroyEffect

## 3.6 Ball System

The ball should include:

Velocity
Direction
Collision response
Speed clamp
Launch behavior
Reset behavior
Trail effect
Audio feedback
Optional skin support from marketplace

Prevent infinite horizontal or vertical movement bugs by enforcing minimum angle corrections.

The ball must never become permanently stuck. Add anti-stall logic.

Anti-stall examples:

If the ball bounces between the same two surfaces too many times, slightly perturb the angle.

If velocity magnitude falls below a threshold, restore minimum speed.

If the ball leaves bounds, reset or subtract life.

If the ball becomes trapped inside geometry, teleport it to a safe reset position.

## 3.7 Paddle System

The paddle should include:

Touch / mouse controlled horizontal movement.

Bounds clamping.

Collision response.

Optional size modifiers.

Optional marketplace cosmetic skins.

Optional powerup effects.

The paddle must feel immediate and predictable.

## 3.8 Level System

Build at least 8 playable levels.

Levels should increase in difficulty by changing:

Brick count
Brick durability
Brick layout complexity
Hazard presence
Ball speed
Moving brick count
Reward opportunities
Required precision

Level examples:

Level 1: Basic tutorial layout
Level 2: Wide brick wall
Level 3: Durable brick introduction
Level 4: Reward brick introduction
Level 5: Hazard brick introduction
Level 6: Shielded or moving brick introduction
Level 7: Mixed challenge
Level 8: Final showcase level

Each level must be completable through normal play.

## 3.9 Level Completion Rules

A level is complete when all required bricks are destroyed.

Not every brick must be required. Bonus bricks may remain optional if intentionally designed.

On completion:

Freeze gameplay.

Play success animation.

Show results screen.

Grant score and currency.

Unlock next level.

Save progress.

Offer next level, replay, marketplace, main menu.

## 3.10 Failure Rules

The player fails when:

Lives reach zero, or

All balls are lost, or

An objective-specific fail condition is met.

On failure:

Freeze gameplay.

Play failure sound / animation.

Show score.

Show currency earned if partial rewards are allowed.

Offer retry.

Offer marketplace.

Offer main menu.

Save any valid currency if design allows it.

# 15. Final Instruction to the Vibe Coder

Build this as a complete final project, not as a minimal classroom assignment.

Prioritize finishing the full vertical slice first, then expand content and polish. Do not overbuild isolated features that do not connect to the player experience. Every system must be visible, testable, documented, and evidenced.

When choosing between technical elegance and submission readiness, preserve both where possible, but never leave a required deliverable missing. A smaller polished feature with documentation and evidence is better than a larger unfinished system.

Use the BrickBlast starter as the foundation, but make the final result clearly original through progression, marketplace customization, multiple brick behaviors, polished UI, sound, persistence, networking, and publication materials.

Final deliverable must include the playable game, source code, documentation, test evidence, architecture diagrams, communication logs, marketplace system, networking proof, publish-ready package, and landscape video trailer.
